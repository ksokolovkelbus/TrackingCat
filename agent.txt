Ниже — review **по присланным файлам**, а не по гипотезам. Важное ограничение: в наборе нет `video_source.py`, `alert_recorder.py`, `audio_alert.py`, `logger_setup.py`, поэтому по ним я оцениваю только контракты и точки интеграции из `main.py`, а не внутреннюю реализацию.

`main.py` действительно является orchestration hub: он загружает конфиг, запускает Zone Editor, создаёт detector/source/tracker/overlay/surface monitor/alert recorder и потом в одном loop читает кадры, запускает detector/tracker, обновляет зоны, пишет overlay, recording, GUI и output video. Это не “плохо”, но это зона будущих регрессий.

Хорошая новость: core-типы уже вынесены в `models.py`: есть `Detection`, `TrackState`, `TrackingPipelineState`, `ZoneType`, `SceneZone`, `TrackLocationState`, `SurfaceEvent`, `ActiveAlertTrack`, `FrameTrackingSummary`, config dataclasses. Это правильная база, её не нужно переписывать.

# 2. Найденные проблемы

## 2.1. Самый важный риск: alert/zone logic ломается в tracker-only режиме

В `DetectThenTrackManager.should_run_detector()` detector может не запускаться каждый кадр: когда `tracker_only_mode_after_confirm=True`, система между YOLO refresh-ами использует OpenCV tracker.

Но `SurfaceMonitor._is_alert_eligible_track()` классифицирует зону только если track `CONFIRMED` и последний YOLO detection был не старше одного кадра:

```python
if track.last_detection_frame <= 0:
    return True
return (frame_index - track.last_detection_frame) <= 1
```

То есть в tracker-only режиме через 2 кадра после YOLO track может перестать участвовать в zone classification, даже если OpenCV tracker уверенно обновляет bbox. Это может давать мигание `current_zone=None`, сброс `active_alert_tracks`, нестабильный continuous alert и нестабильную запись инцидентов.

Это сейчас, на мой взгляд, главный кандидат на “почему система не как швейцарские часы”.

**Что менять:** `surface_monitor.py`.

**Безопасная политика:**

* `CONFIRMED` track классифицировать всегда по текущему bbox.
* `HELD` track можно либо классифицировать по последнему bbox с ограничением, либо не давать ему генерировать новые entry events, но разрешить удерживать active alert короткое время.
* Не привязывать zone classification к свежести YOLO detection; свежесть YOLO должна влиять на confidence/policy, а не обнулять location state.

---

## 2.2. `SurfaceMonitor` возвращает candidate events как будто они emitted

В `SurfaceMonitor.update()` событие добавляется в `events`, а уже потом `_handle_surface_event()` может его подавить из-за cooldown/global interval/per-track interval.

Сейчас результат называется `surface_events`, но фактически там могут быть события, по которым звук не проигран. Это опасно, если `AlertRecorder` или будущая логика начнёт считать `surface_events` именно “реальными alert events”.

**Что менять:** `surface_monitor.py`, `models.py`.

**Лучше:** вернуть `AlertDecision` или минимум разделить:

```python
emitted_surface_events: list[SurfaceEvent]
suppressed_surface_events: list[SuppressedSurfaceEvent]
```

---

## 2.3. Нет hysteresis на вход в surface/restricted

Сейчас есть `min_floor_frames_before_alert`, то есть защита “перед входом кот должен побыть на floor”. В YAML для webcam safe это `4` кадра.

Но нет симметричного условия “кот должен быть в surface/restricted N кадров перед alert”. Поэтому jitter центра bbox на границе зоны может дать entry event на первом же кадре.

**Что менять:** `models.py`, `config.py`, `surface_monitor.py`, YAML.

Добавить:

```yaml
surface_alert:
  min_zone_frames_before_alert: 2
  min_zone_dwell_seconds_before_alert: 0.15
```

Для `restricted` можно оставить более агрессивно:

```yaml
restricted_min_zone_frames_before_alert: 1
surface_min_zone_frames_before_alert: 2
```

---

## 2.4. Zone classification для track и alert использует разные правила

В `SceneZoneClassifier.classify_track()` используется foot point + bbox overlap threshold. В `classify_alert_track()` используется только `track_crosshair_point`, то есть центр/crosshair.

Это может быть нормально, если так задумано, но operationally это риск:

* overlay/track location может говорить одно;
* alert logic может видеть другое;
* на границе зоны центр bbox ещё вне restricted, хотя лапы/тело уже внутри.

Плюс `bbox_overlap_ratio()` использует bounding box зоны, а не реальное polygon intersection. Для polygon-зон сложной формы это может давать ложный overlap.

**Что менять:** сначала тесты в `tests/test_zones.py`, потом `zones.py`.

Не обязательно сразу делать polygon intersection. Но надо явно зафиксировать expected behavior.

---

## 2.5. `SurfaceMonitor` смешивает 4 ответственности

В одном классе сейчас:

1. zone state per track;
2. alert event policy;
3. cooldown/suppression;
4. audio side effects;
5. overlay message state.

Это видно по тому, что `SurfaceMonitor` принимает `SceneZoneClassifier`, `SurfaceAlertConfig`, `logger`, `AudioAlertPlayer`, хранит `_track_states`, `_last_global_sound_ts`, `_overlay_message`, вызывает `play()`/`start_continuous()`/`stop_continuous()`.

Это ещё не катастрофа, но именно здесь будут самые болезненные регрессии.

**Не надо сразу выносить в 5 файлов.**
Первый безопасный шаг — внутри `surface_monitor.py` разделить приватно:

```text
ZoneStateUpdater
AlertPolicy
AlertActuator
```

И только потом, если станет удобно, вынести в отдельные модули.

---

## 2.6. Config layer хороший, но `bool(...)` опасен

В `config.py` многие поля строятся через `bool(value)`. Для настоящих YAML boolean это нормально, но если в YAML случайно появится строка `"false"`, Python даст `True`.

Пример зоны: `enabled=bool(scene_zones_data.get(...))`.

CLI защищён `parse_bool`, но YAML — нет.

**Что менять:** `config.py`.

Добавить `_coerce_bool(value, field_name)`:

* bool → как есть;
* int 0/1 → bool;
* string `"true"/"false"/"yes"/"no"/"on"/"off"` → parse;
* иначе `ConfigError`.

Это low-risk, но обязательно добавить tests.

---

## 2.7. CPU performance: не хватает stage timings

Сейчас есть `FPSMeter`, но он меряет общий FPS между итерациями.

Не видно, где реально bottleneck:

* source read/decode;
* resize;
* YOLO;
* tracker update;
* surface monitor;
* overlay;
* alert recording;
* `cv2.imshow`;
* video writer.

Без этого оптимизация будет гаданием. Особенно для iPhone MJPEG и ESP32 snapshot.

**Что менять:** `main.py`, возможно новый `performance.py`.

---

## 2.8. OpenCV tracker backend order не конфигурируется

`OpenCvFrameTracker` выбирает backend в порядке `CSRT`, `KCF`, `MIL`.

CSRT обычно точнее, но тяжелее на CPU. Для “visual tracking mode” или слабой машины KCF может быть практичнее. Сейчас это нельзя переключить из YAML.

**Что менять:** `models.py`, `config.py`, `tracker.py`, YAML presets.

Добавить:

```yaml
tracking:
  frame_tracker_backend: KCF
```

Или:

```yaml
tracking:
  frame_tracker_backends: [KCF, CSRT, MIL]
```

---

## 2.9. `main.py` перегружен, но переписывать его не надо

Текущий loop делает всё в одном месте: read, resize, FPS, detector/tracker, surface monitoring, coordinate logging, zone drawing, status, alert recorder, output writer, GUI.

Это не повод “переписать всё”. Но стоит вынести безопасные функции:

```python
read_frame_step()
tracking_step()
alert_step()
render_step()
recording_step()
```

Или один объект:

```python
TrackingRuntime.process_frame(frame, timestamp) -> FrameResult
```

Но это этап после тестов.

---

## 2.10. Absolute path в alert recording config

В `webcam_aiming_safe.yaml`:

```yaml
alert_recording:
  output_dir: /home/kelbus/PycharmProjects/TrackingCatWIFI/alert_video
```

Это работает локально, но снижает переносимость и может ломать запуск из другого checkout.

**Что менять:** YAML + config path resolution.

Лучше:

```yaml
alert_recording:
  output_dir: alert_video
```

И резолвить относительно project root/config dir/current working dir по явному правилу.

## Улучшения на 1 день

### 4.5. Исправить `SurfaceMonitor._is_alert_eligible_track`

**Что менять:** `surface_monitor.py`.

Текущая проблема — eligibility завязана на `last_detection_frame`. Нужно заменить на policy примерно такого вида:

```python
def _is_zone_classification_eligible(track: Track) -> bool:
    return track.state in {TrackState.CONFIRMED, TrackState.HELD}
```

А для **генерации новых entry alerts** использовать более строгую policy:

```python
def _can_emit_new_entry_alert(track: Track, frame_index: int) -> bool:
    if track.state != TrackState.CONFIRMED:
        return False
    return True
```

То есть:

* классифицировать зону можно по текущему bbox tracker-а;
* создавать новый alert лучше только для `CONFIRMED`;
* `HELD` можно использовать для удержания текущего continuous alert короткое время, но не для нового entry.

**Зачем:** не обнулять zone state в tracker-only mode.

**Эффект:** максимальный прирост стабильности.

**Сложность:** средняя.

**Риск:** средний.

**Тест обязателен:** да.

### 4.6. Разделить candidate event и emitted event

**Что менять:** `surface_monitor.py`, `models.py`, возможно `main.py`.

Сейчас `_maybe_build_surface_event()` создаёт event, потом `_handle_surface_event()` может его подавить.

Лучше:

```python
@dataclass(frozen=True)
class AlertDecision:
```

```python
SurfaceMonitorResult(
    emitted_surface_events=[...],
    suppressed_surface_events=[...],

```

**Зачем:** downstream code не должен гадать, был ли alert реально emitted.

**Эффект:** чище incident recording и будущие тесты.

**Сложность:** средняя.

**Риск:** средний.

**Тест обязателен:** да.

### 4.12. Подготовить `FramePacket` / source metadata

```python
@dataclass(frozen=True)
class FramePacket:
    frame: np.ndarray
    source_id: str
    frame_index: int | None
    captured_at_ts: float
    received_at_ts: float
    read_ms: float
    decode_ms: float | None = None
    stale: bool = False
```

**Зачем:** ESP32 snapshot и iPhone MJPEG имеют latency/stale frame проблемы.

**Эффект:** лучше диагностика и стабильность source layer.

**Сложность:** средняя/высокая.

**Риск:** средний.

**Тест:** нужен fake source.

# 7. Что нельзя трогать без тестов

1. `SurfaceMonitor._is_alert_eligible_track()` и вся логика `_maybe_build_surface_event()` / `_handle_surface_event()`
   Это напрямую влияет на реальные алерты.

2. `SceneZoneClassifier.classify_point()` / `_zone_type_rank()`
   Это safety-critical priority зон. Сейчас priority documented и должен быть стабильным.

3. `MultiCatTracker._handle_unmatched_track()`
   Там lifecycle `CONFIRMED -> HELD -> LOST -> REMOVED`; любое изменение может сломать reacquire и alert stability.

4. `ZoneEditor._save_zones()`
   Сейчас он явно сохраняет normalized zones и отключает editor mode. Это уже исправленная зона риска.

5. Coordinate conversion в `zones.py`
   Нельзя менять без normalized/pixels tests.

6. `AlertRecorder` integration в `main.py`
   Файл не прислан, но по main видно, что recording зависит от `active_alert_tracks` и overlay state. Любое изменение surface events может повлиять на записи.

# 8. Что даст максимальный эффект быстро

## Максимальная стабильность

1. Исправить `SurfaceMonitor` так, чтобы zone state не зависел от свежего YOLO detection.
2. Разделить emitted/suppressed events.
3. Добавить entry hysteresis.
4. Добавить sequence regression tests.
5. Добавить source/performance diagnostics.

## Максимальная CPU-производительность

1. Stage timing telemetry — сначала измерить.
2. Для tracker-only modes сделать KCF первым backend или configurable backend.
3. Увеличить `detector_interval_while_tracking` там, где это безопасно.
4. Настроить `imgsz` и resize по фактическому FPS.
5. Убрать per-frame INFO logs.
6. Позже — latest-frame source для iPhone/MJPEG и ESP32 snapshot, чтобы не обрабатывать старые кадры.

## Что сейчас вероятнее всего мешает “швейцарским часам”

Главный кандидат — **разрыв между tracker-only pipeline и alert/zone pipeline**: tracker может продолжать вести bbox, но `SurfaceMonitor` перестаёт классифицировать зону, если YOLO detection не свежий. Это создаёт нестабильность именно в тех режимах, где CPU-only оптимизация должна полагаться на tracker между detector refresh-ами.

Второй кандидат — отсутствие stage timing/source metadata: без этого трудно понять, где лаг — YOLO, камера, decode, overlay или recorder.

Третий — alert event semantics: suppressed event сейчас может выглядеть как обычный event.

# 9. Итоговая рекомендация

Архитектура проекта **не требует переписывания**. Она уже имеет нормальные границы: `detector`, `tracker`, `zones`, `surface_monitor`, `overlay`, `config`, `zone_editor`. Главная задача — не “сделать красиво”, а стабилизировать стыки:

```text
tracker -> zone state -> alert decision -> recorder/audio/overlay
```

Я бы делал первый безопасный refactor в таком порядке:

1. Tests для `SurfaceMonitor` tracker-only behavior.
2. Fix eligibility/classification policy.
3. Tests для emitted/suppressed events.
4. Разделить emitted/suppressed.
5. Добавить stage timing.
6. Добавить `_coerce_bool`.
7. Добавить entry hysteresis.
8. Потом уже tracker backend config и source metadata.

Это даст лучший баланс риск/польза для живого CPU-only проекта.
